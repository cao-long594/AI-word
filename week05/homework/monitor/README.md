## 一、已完成功能

### 1. 支持通过 JSON 配置文件加载监控目标

程序可以从外部 JSON 文件读取待探测目标，配置项包括：

- `name`：目标名称
- `address`：目标地址
- `protocol`：探测协议
- `expect`：期望结果
- `retry_count`：失败后的重试次数



### 2. 支持多种探测协议



- **HTTP / HTTPS 探测**：发送 GET 请求并判断响应结果；
- **TCP 探测**：尝试建立 TCP 连接，判断端口是否可达。

程序会根据配置中的 `protocol` 字段自动选择探测方式。

### 3. 实现高并发探测



在 `ProbeAll()` 中，程序会遍历所有目标，并为每个目标启动一个 goroutine：

- 每个 goroutine 负责一个目标的完整探测流程；
- 所有目标并发执行，而不是顺序阻塞；
- 主 goroutine 通过 `WaitGroup` 等待所有子任务结束；
- 最终统一汇总探测结果。


### 4. 支持超时控制

为了避免某些目标长时间无响应导致程序卡住：

- HTTP 探测使用 `context.WithTimeout` 控制请求超时；
- TCP 探测使用 `net.DialTimeout` 控制连接超时。

这样即使目标不可达或响应很慢，也能在指定时间内结束本次探测。

### 5. 支持失败重试

每个目标都支持配置 `retry_count`。

当探测失败时，程序会按配置进行重试：

- 只要某次探测成功，就提前结束重试；
- 如果全部失败，则记录最终失败结果。

该机制可以降低因为网络瞬时抖动带来的误判。

### 6. 支持多种期望判断方式

针对 HTTP 探测，支持以下判断逻辑：

- `expect = "200"`：要求 HTTP 状态码为 200；
- `expect = "contains:关键字"`：要求响应体包含指定内容；
- `expect = "fail"`：要求请求失败；
- `expect = "timeout"`：要求请求超时。



### 7. 支持终端统计报告输出

程序执行完成后会输出较完整的探测报告，包括：

- 总目标数、成功数、失败数、成功率；
- 最小耗时、平均耗时、P50 / P90 / P99、最大耗时；
- 最慢目标列表；
- 失败目标列表；
- 彩色终端输出，便于快速查看结果。

### 8. 支持命令行参数

程序入口支持：

- `-config`：指定配置文件路径；
- `-timeout`：指定单次探测超时时间；
- `-v`：开启详细模式，实时打印每个目标的探测结果。

### 9. 编写了单元测试

项目测试覆盖了以下主要场景：

- 配置文件加载成功 / 失败；
- HTTP 200 成功；
- 响应体包含关键字成功 / 失败；
- `expect=fail` 场景；
- `expect=timeout` 场景；
- TCP 探测成功 / 失败；
- 不支持协议场景；
- 并发探测返回完整结果；
- 重试后成功场景。

## 二、开发中遇到的问题与解决方案

### 问题 1：如何实现“同时探测多个目标”

**问题描述：**
刚开始如果直接用普通 `for` 循环依次探测，程序会一个目标接一个目标执行，总耗时较长，无法体现高并发探测。

**解决方法：**
我把每个目标放到独立 goroutine 中执行，并使用 `sync.WaitGroup` 统一等待。这样多个目标能够并行探测，整体耗时明显降低。

---

### 问题 2：多个 goroutine 同时写入结果会产生并发安全问题

**问题描述：**
多个 goroutine 同时向 `results` 切片追加结果时，可能出现数据竞争，导致结果丢失或程序行为异常。

**解决方法：**
在写入 `results` 时使用 `sync.Mutex` 加锁，保证同一时间只有一个 goroutine 可以修改结果切片，从而保证数据安全。

---

### 问题 3：目标无响应时程序可能卡住

**问题描述：**
如果某些目标响应很慢或者根本不可达，程序可能会一直等待，影响整批探测效率。

**解决方法：**
HTTP 请求使用 `context.WithTimeout`，TCP 连接使用 `net.DialTimeout`，为每次探测设置明确超时，这样慢目标不会无限阻塞。

---

### 问题 4：对于探测状态的判断

**问题描述：**

- 对于无效服务：期望结果是`"fail"`
- 对于慢速响应模拟期望结果有两种情况：5s内为timeout，其他为200 OK；但是expect只能选择一种比如`"timeout"`

**解决方法：**
程序通过 `expect` 字段灵活控制探测结果的判定逻辑，支持多种验证场景：

1. **期望失败 (`expect: "fail"`)**：
    - 判断逻辑：
        - HTTP 探测：请求失败（超时/连接错误）→ 成功 ✅；收到正常响应 → 失败 ❌

2. **期望超时 (`expect: "timeout"`)**：
   - 判断逻辑：
      - timeout<=5s时，在超时时间内未收到响应 → 成功 ✅（符合超时预期）
      - timeout>5s时，在超时时间内收到 200 响应 → 成功 ✅（服务响应快于预期）
      - 其他情况 → 失败 ❌
```go
case expect == "timeout":
    if resp.StatusCode == 200 {
        res.Success = true
    } else {
        res.Success = false
        res.Error = "expected timeout or 200, but got status: " + fmt.Sprint(resp.StatusCode)
    }
```
